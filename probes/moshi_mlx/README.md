# Slice 2B — Moshi/MLX viability probe

This directory holds the viability probe that gates Slice 2C (the
real `audio.realtime.session` adapter). The probe answers a single
**scaffold-level** question:

> Can a Moshi-on-MLX implementation produce **correctly-shaped
> streaming audio events** on macOS-arm64 within Octomil's latency /
> memory budgets AND map cleanly onto the existing `oct_session_*`
> ABI?

It does NOT prove "produces audible speech" — the streaming input
is all-zero silence, by design (deterministic, network-free, no
prompt corpus). Speech-generation viability is a separate
operator-driven step (run the canonical `moshi_mlx.run_inference`
against a non-silence prompt + listen / measure RMS / WER). The
probe's GREEN verdict means "ABI plumbing + tensor shapes + timing
budgets + artifact pinning all pass"; it does NOT mean "Moshi
outputs intelligible speech." Codex R11 honesty fix.

If GREEN → Slice 2C builds the C++ adapter that wraps moshi-mlx.
If RED → fall back to one of the candidates listed in `manifest.toml`.
If ABI_DELTA_REQUIRED → STOP, debate the gap before any code change.

## Scope (NOT done in this probe)

- No daemon (slice 3b territory).
- No iOS framework changes (slice 4).
- No public Octomil SDK surface change.
- No Python realtime oracle (the probe never advertises
  `audio.realtime.session` from `oct_runtime_capabilities`).
- No ABI changes. If the probe uncovers an ABI gap, it stops with
  a `ABI_DELTA_REQUIRED` verdict and the gap is debated BEFORE any
  code change.

## Files

| File                 | Purpose                                                                                                           |
| -------------------- | ----------------------------------------------------------------------------------------------------------------- |
| `manifest.toml`      | Pinned upstream commits, weight hashes, license, budgets, and the design-time ABI mapping table. Source of truth. |
| `probe.py`           | Reads the manifest, verifies the local artifact root, runs streaming inference, and writes `probe-results.json`.  |
| `event_mapping.md`   | Long-form ABI mapping rationale (why each Moshi output fits an existing `oct_event_*` payload).                   |
| `manifest.lock.toml` | Auto-generated by the probe — captures the AS-RUN versions and hashes for reproducibility.                        |
| `probe-results.json` | Auto-generated. The artifact the debate consumes.                                                                 |

## Running the probe

```bash
# 1. Install the optional deps (macOS-arm64 only).
uv sync --extra moshi-probe

# 2. Pre-stage the weights into a local artifact root. Sizes are
#    several GB; the probe will NOT auto-download. Note: kyutai/
#    moshiko-mlx-q4 bundles the Mimi tokenizer and LM config in the
#    same HF repo — there is NO separate `kyutai/mimi` directory.
mkdir -p ~/octomil-artifacts/moshi-v0.2/moshiko_mlx_q4
huggingface-cli download kyutai/moshiko-mlx-q4 \
    --local-dir ~/octomil-artifacts/moshi-v0.2/moshiko_mlx_q4

# 3. Run the probe. First run requires OCTOMIL_PROBE_BOOTSTRAP=1
#    to record AS-RUN sha256 hashes into manifest.lock.toml; without
#    it, acceptance #2 RED's because manifest.toml ships empty pins
#    by design (we cannot pre-commit hashes for weights we have not
#    downloaded). Subsequent runs verify against the lock.
OCTOMIL_PROBE_BOOTSTRAP=1 uv run python probes/moshi_mlx/probe.py \
    --artifact-root ~/octomil-artifacts/moshi-v0.2 \
    --output probes/moshi_mlx/probe-results.json
```

The probe exits 0 on `GREEN`, 1 on `RED` or `ABI_DELTA_REQUIRED`.

## Acceptance gates (mirrored in `manifest.toml`)

1. MLX optional dependency installs cleanly on macOS-arm64.
2. Exact Moshi repo / commit / license / weight hashes pinned and
   verified at runtime.
3. `moshi_mlx.models.Lm` + `rustymimi.Tokenizer` instantiate from a
   local artifact path (no Hugging Face network call during the
   probe). Note: the canonical inference path is `rustymimi.Tokenizer`,
   NOT `moshi_mlx.models.mimi.Mimi` — the latter is for training/
   export. Slice 2C's C++ adapter wraps the rustymimi path.
4. Streaming loop produces ≥ 8 `audio_chunk`-shaped events whose
   payload fits `oct_event_t.data.audio_chunk` field-for-field.
5. Measurements within budget (see `manifest.toml [probe.budgets]`):
   cold_open ≤ 8s, warm_open ≤ 1.5s, first_audio ≤ 1.2s,
   real_time_factor ≤ 1.0 (avg compute-per-chunk / 80 ms),
   compute_per_chunk_max_ms ≤ 160 (underrun-risk hard cap),
   peak_rss ≤ 6 GB, gpu_active ≥ 30% (operator-supplied via
   OCTOMIL_PROBE_GPU_PCT). compute_per_chunk_p99_ms,
   cancel_to_silent_python_proxy_ms, and audio_chunk_validation
   signal stats (output_rms / output_peak_abs) are recorded
   **informationally**; they do NOT gate GREEN.
6. Every Moshi output column maps to a defined `oct_event_*` payload
   AND every required input fits `oct_session_config_t`.

## Verdict semantics

- `GREEN` — every gate passes. Output: prepare Slice 2C.
- `RED` — at least one gate fails (timing, RSS, GPU, etc.). Output:
  bring measurements + fallback recommendation to debate.
- `ABI_DELTA_REQUIRED` — Moshi needs a field that doesn't exist on
  `oct_session_config_t` or `oct_event_t`. Output: STOP. Run the
  engineering-debate workflow on the ABI delta proposal BEFORE any
  code change. The header is the contract; growing it without
  consensus would defeat the slice 2A audit trail.
